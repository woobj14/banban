2 기능별 냉정 평가

각 기능: 점수 · 잘된 점 · 날카로운 단점 · 그룹 합성 후기

① 서술형 DNA 7.8 / 10

잘됨: 내가 공부한 노트에서만 나오는 서술형 = 시중 문제집이 못 하는 진짜 차별점. 스캐폴딩 3단계로 영작 공포 완화, "이렇게 바꾸면 더 좋아요" 첨삭이 호평.

  • AI 채점 신뢰도: 중학생 영작은 철자·관사·시제 변형이 무한 → LLM이 정답을 오답 처리(거짓 음성)하면 학생 신뢰가 즉시 붕괴. S1·S3에서 가장 치명적.
  • 비용: 채점 1건 = LLM 1콜. 1,000명이 하루 5문제면 일 5,000콜. 단일 키 공유 시 레이트리밋·비용 폭증.
  • 지연: Streamlit rerun + LLM 왕복으로 채점 버튼 후 3~6초 정적. S5는 "왜 멈춰요?"로 이탈.

S3 준호 "맞게 썼는데 틀렸대요. 모범답안이랑 뜻 똑같은데… 이럴 거면 안 해요."

T3 이쌤 "방향은 최고예요. 근데 채점이 한 번이라도 틀리면 학부모 민원 들어와요. 부분점수 근거를 보여줘야 신뢰합니다."

② 노트 라이브러리 Supabase 이전 5.5 / 10

잘됨: 로컬↔배포 동기화 + 재배포에도 노트 영속 = 운영의 기본기를 드디어 갖춤. 캐싱으로 클릭 반응도 개선.

  • 🚨 멀티테넌시 부재 (구조적 결함): 현재 notes는 owner 필터가 없어 전체 공유. 즉 박쌤이 만든 노트를 강쌤·이쌤이 그대로 봄. 1인 공부방 교사의 핵심 자산(직접 만든 교재)이 경쟁 교사에게 노출 → 유료 전환의 근간을 파괴.
  • 전체 공유의 2차 문제: 노트 수천 개가 한 목록에 → 학생이 남의 반 노트를 봄, 필터 지옥.

T2 박쌤 "제가 밤새 만든 3과 교재가 왜 옆 학원 목록에 떠요? 이러면 절대 결제 안 하죠. 당장 내 노트만 보이게 해주세요."

T5 윤쌤 "콘텐츠 자산화가 셀링포인트인데 자산이 공유라니… 신뢰가 안 가요."

③ 로그인 첫 화면 → 내 학습현황 6.8 / 10

잘됨: 학생은 들어오자마자 내 진도·복습이 보여 학습 동기 ↑. 결제 배너부터 보던 첫인상 문제 해소.

  • 신규 선생님 빈 화면: 학생 0명·노트 0개인 가입 직후 선생님은 텅 빈 대시보드를 첫 화면으로 봄 → "뭘 해야 하지?" 막막함. 선생님은 첫 할 일이 노트 만들기인데 그게 안 보임.
  • 역할 무분별: 학생·선생님 모두 같은 첫 화면 → 선생님에겐 "노트 만들기" 온보딩이 더 적합.

T2 박쌤(신규) "로그인했더니 빈 표만 있어요. 첫날엔 '노트부터 만드세요' 같은 안내가 낫겠는데요."

④ PRO 업그레이드 CTA 미니멀화 6.5 / 10

잘됨: 깨진 배너 제거로 사이드바가 단정해짐. 첫인상 퀄리티 회복.

  • 전환 약화 위험: 너무 약해진 한 줄 링크는 배너 블라인드니스로 무시됨. 기능 잠금 지점의 upgrade_banner()가 실제로 잘 노출되는지 검증 안 되면 매출 동선이 통째로 사라질 수 있음.
  • 가치 제안 부재: "PRO로 업그레이드"만으론 왜 사야 하는지 모름. "서술형 무제한" 같은 핵심 후킹 1개는 남겨야.

T4 최쌤 "깔끔해서 좋은데… 솔직히 PRO가 뭘 더 주는지 한눈에 안 와요."

⑤ 학생 가입 개편 (이메일·연락처 선택 + 코드 선택) 7.0 / 10

잘됨: 선생님 코드 없이도 개인 가입 가능 → 무료 유입 확대. 선생님 코드 시 plan 상속 = 반 전체 커버.

  • 계정 복구 사각: 이메일이 선택이라 비번 분실 시 복구 불가한 학생 대량 발생 가능(S3·S4). CS 부담 ↑.
  • 수익 모델 모호: 무료 개인 학생이 늘면 AI 비용만 증가하고 STUDENT 전환은 미지수. 무료↔유료 경계 가치가 약함.

S4 지우 학부모 "아이가 비번을 까먹었는데 이메일을 안 넣어서 못 찾는대요."

⑥ Polar 결제 자동화 (Webhook) 5.8 / 10

잘됨: 결제 완료 → plan 자동 반영 파이프라인 구축. 수동 변경 탈피.

  • 🚨 이메일 매칭 실패 (구조적 결함): 학생 로그인은 가상 이메일(username 기반)인데, webhook은 결제자 이메일로 profiles.email/contact_email을 매칭. 둘이 다르면 결제해도 plan이 안 올라감 → "돈 냈는데 PRO 안 됨" 최악의 경험.
  • 서명 검증 미완: HMAC 검증이 헤더 존재 확인 수준 → 위조 webhook으로 무단 plan 승격 가능.

S2 서연 "STUDENT 결제했는데 아직도 잠겨 있어요. 환불해주세요." — 가장 위험한 문장

⑦ 성능 — 캐싱 + Render 무료 콜드스타트 4.9 / 10

잘됨: 노트 조회 캐싱(120s)으로 클릭 단위 지연은 개선됨.

  • 🚨 콜드스타트: 무료 플랜은 15분 유휴 후 sleep → 첫 접속 30초~1분 백지. 학원 수업 시작 시 30명이 동시에 1분을 기다림 → 수업 흐름 파탄.
  • Streamlit rerun: 매 상호작용 전체 재실행. 저사양 폰(S3 다수)에서 체감 지연 상존.

T1 강쌤 "수업 종 치고 앱 켰는데 1분을 멍하니 기다려요. 이건 현장에서 못 씁니다."

4 🚨 지금 막아야 할 치명 리스크 3

아래는 추정이 아니라 코드에서 확인된 구조적 사실. 기능 추가보다 우선.

① 노트 멀티테넌시 부재 — 교사 자산 전체 공유

library.py가 owner 필터 없이 전체 notes를 반환. 선생님이 만든 교재가 다른 모든 선생님·학생에게 노출됨. 유료 전환의 핵심 동기(내 콘텐츠 자산화)를 정면으로 무너뜨림. → owner_id(이미 컬럼 존재) 기준 필터 + RLS 재설계 필요.

② Polar webhook 이메일 매칭 실패 — "결제했는데 PRO 안 됨"

학생은 가상 이메일로 로그인하는데 webhook은 결제 이메일로 매칭 → 불일치 시 plan 미반영. 결제는 됐는데 혜택이 안 열리는 환불·신뢰 붕괴 시나리오. → 결제 시 checkout에 user_id를 메타데이터로 실어 보내 ID 기준 매칭으로 전환 + HMAC 서명 검증 완성.

③ 무료 플랜 콜드스타트 — 수업 현장 1분 백지

Render 무료는 유휴 후 sleep. 학원 단체 접속 시 동시 대기. B2B(학원) 채택의 결정적 장벽. → Starter 유료 전환(근본) 또는 uptime 핑(임시). 운영 비용으로 처리할 1순위.

5 ✅ 분명히 잘한 것

서술형 DNA — 진짜 해자(moat)
